CPIT 252 — Design Patterns

Complete exam cheat sheet — intent · usage · implementation · code smells · MVC
What are design patterns?
Proven, reusable solutions to commonly occurring problems in software design.
Not copy-paste code — they are ideas / templates you apply to your situation.
Solve complex problems
Give you a tested solution instead of inventing one from scratch.
Improve readability
Other developers recognize the pattern and understand your code faster.
Loose coupling
Add new features without breaking existing code. (Open/Closed Principle)

Creational patterns
SingletonCreational
Intent
Only ONE instance of a class exists in the whole app. Everyone shares it.
When to use
Database connection · App logger · Config settings — things shared and created only once.
Implementation
class Database { private static Database instance; private Database() { } // private = no one can do new Database() public static Database getInstance() { if (instance == null) instance = new Database(); return instance; } } // Usage Database db1 = Database.getInstance(); Database db2 = Database.getInstance(); // db1 == db2 → TRUE, same object
PrototypeCreational
Intent
Clone / copy an existing object instead of building a new one from scratch.
When to use
Game character duplication · Any object that is expensive to create.
Implementation
class Character implements Cloneable { String name; int health; public Character clone() { return (Character) super.clone(); } } Character c1 = new Character("Ali", 100); Character c2 = c1.clone(); // cheap copy c2.name = "Ali2"; // c1.name still "Ali"
BuilderCreational
Intent
Build complex objects step by step. Same building process, different results.
When to use
Burger with many optional items · HTTP request with many optional fields.
Implementation
Burger b = new Burger.Builder() .withBun("sesame") .withPatty("beef") .withCheese() .withSauce("bbq") .build(); // Builder holds each field. build() returns final object.
Factory MethodCreational
Intent
Parent says "create a transport" — subclass decides if it's a Truck or Ship.
When to use
When you don't know in advance which type to create. Easy to add new types later.
Implementation
abstract class Logistics { abstract Transport createTransport(); // factory method void planDelivery() { createTransport().deliver(); } } class RoadLogistics extends Logistics { Transport createTransport() { return new Truck(); } } class SeaLogistics extends Logistics { Transport createTransport() { return new Ship(); } }
Abstract FactoryCreational
Intent
Create a whole FAMILY of related objects. Swap the factory = swap everything at once.
When to use
UI toolkit: WindowsFactory → Windows button + checkbox. MacFactory → Mac button + checkbox.
Implementation
interface UIFactory { Button createButton(); Checkbox createCheckbox(); } class MacFactory implements UIFactory { Button createButton() { return new MacButton(); } Checkbox createCheckbox() { return new MacCheckbox(); } } // Swap MacFactory ↔ WindowsFactory = swap whole UI kit
Structural patterns
AdapterStructural
Intent
Make two incompatible interfaces work together. Like a power plug adapter.
When to use
Client sends city name. Library expects GPS coordinates. Adapter converts city → lat/lon → delegates.
Implementation
class FajrTimeAdapter implements FajrTimeCity { private FajrTimeGeo geo; FajrTimeAdapter(FajrTimeGeo geo) { this.geo = geo; } public String getFajrTime(String city) { double lat = getLatitude(city); double lon = getLongitude(city); return geo.getFajrTime(lat, lon); // delegate to adaptee } }
BridgeStructural
Intent
Separate "what it is" from "how it works" so both can change independently.
When to use
Shape (Circle/Square) + Color (Red/Blue) — combine freely without a class explosion.
Implementation
abstract class Shape { protected Color color; // holds implementation Shape(Color c) { color = c; } abstract void draw(); } class Circle extends Shape { void draw() { System.out.println("Circle in " + color.fill()); } } // Circle+Red, Circle+Blue, Square+Red... all work with 2+2 classes
CompositeStructural
Intent
Treat a single object and a group of objects the SAME way. Tree structure.
When to use
File system: File and Folder both have getSize(). Folder's size = sum of its children's sizes.
Implementation
interface FileSystemItem { int getSize(); } class File implements FileSystemItem { int size; public int getSize() { return size; } } class Folder implements FileSystemItem { List<FileSystemItem> children = new ArrayList<>(); public int getSize() { return children.stream().mapToInt(FileSystemItem::getSize).sum(); } }
DecoratorStructural
Intent
Add new behavior to an object by WRAPPING it — without changing its class. Stack layers.
When to use
Coffee: SimpleCoffee → wrap with Milk → wrap with Sugar. Each wrapper adds cost + description.
Implementation
abstract class CoffeeDecorator implements Coffee { protected Coffee coffee; CoffeeDecorator(Coffee c) { coffee = c; } } class MilkDecorator extends CoffeeDecorator { public double getCost() { return coffee.getCost() + 0.5; } public String getDescription() { return coffee.getDescription() + ", Milk"; } } // Usage: new SugarDecorator(new MilkDecorator(new SimpleCoffee()))
FacadeStructural
Intent
One simple interface hides a complex subsystem. One call does everything.
When to use
HomeTheater.watchMovie() → dims lights + turns on TV + sets sound. Client calls ONE method.
Implementation
class HomeTheaterFacade { TV tv; SoundSystem sound; Lights lights; void watchMovie() { lights.dim(20); tv.on(); tv.setInput("HDMI"); sound.on(); sound.setVolume(30); } } // Client only needs: theater.watchMovie();
FlyweightStructural
Intent
Save memory by SHARING the common (intrinsic) state across thousands of objects.
When to use
Forest: 1M trees but only 2 tree types. Type/texture = shared (intrinsic). x,y position = unique (extrinsic).
Implementation
class TreeType { String name, texture; void draw(int x, int y){...} } // shared class TreeFactory { static Map<String,TreeType> cache = new HashMap<>(); static TreeType get(String name, String tex) { return cache.computeIfAbsent(name, k -> new TreeType(name,tex)); } } class Tree { int x,y; TreeType type; } // unique position per tree
ProxyStructural
Intent
A stand-in that controls access to the real object. Same interface. Adds validation / lazy-load / cache.
When to use
SpeechToText Proxy: checks API key first. If valid → create real service + delegate. If invalid → deny access.
Implementation
class SpeechProxy implements SpeechToTextService { private RealSpeechToText real; private String apiKey; public String convertSpeechToText(String file) { if (!isValid()) return "Access denied"; if (real == null) real = new RealSpeechToText(); // lazy init return real.convertSpeechToText(file); } private boolean isValid() { return apiKey != null && !apiKey.isEmpty(); } }
Behavioral patterns
Chain of ResponsibilityBehavioral
Intent
Pass a request along a chain. Each handler processes it OR passes it to the next one.
When to use
Support tickets: Level1 → Level2 → Level3 Manager. Each handles what it can.
Implementation
abstract class Handler { protected Handler next; void setNext(Handler h) { next = h; } abstract void handle(int level); } class Level1 extends Handler { void handle(int level) { if (level == 1) System.out.println("L1 handled"); else if (next != null) next.handle(level); // pass forward } }
CommandBehavioral
Intent
Wrap a request as an object so you can queue it, log it, or UNDO it.
When to use
Text editor Undo/Redo: every action = Command object. Stack stores history. Undo = pop + call undo().
Implementation
interface Command { void execute(); void undo(); } class WriteCommand implements Command { TextEditor editor; String text; void execute() { editor.write(text); } void undo() { editor.deleteLast(text.length()); } } // Invoker stores Stack<Command>. run() pushes. undo() pops.
ObserverBehavioral
Intent
Subject notifies ALL observers automatically when its state changes. One-to-many.
When to use
Player scores points → all Fans notified. Model changes → all Views refresh (this is MVC!).
Implementation
// Subject interface: attach(), detach(), notifyObservers() // Observer interface: update(Player p) class Player implements Subject { List<Observer> observers = new ArrayList<>(); void setCurrentPoints(int pts) { this.currentPoints = pts; if (pts > sessionAvg) notifyObservers(); // auto-notify } void notifyObservers() { for (Observer o : observers) o.update(this); } } class Fan implements Observer { void update(Player p) { System.out.println(p.getName() + " exceeded avg!"); } }
StrategyBehavioral
Intent
Family of interchangeable algorithms. Pick one at runtime. No if/else chain needed.
When to use
Payment: CreditCard / PayPal / Cash. File parser: JSON / XML / CSV. Sorting algorithms.
Implementation
interface PaymentStrategy { void pay(double amount); } class CreditCard implements PaymentStrategy { void pay(double amount) { System.out.println("Paid by card: " + amount); } } class OrderProcessor { PaymentStrategy strategy; void processOrder(double total) { strategy.pay(total); } // no if/else! } // Swap: processor.strategy = new PayPalPayment();
StateBehavioral
Intent
Object changes its behavior when its internal state changes. Same method, different result.
When to use
Vending machine: IdleState vs HasMoneyState. Same pressButton() call behaves differently.
Implementation
interface VendingState { void insertCoin(VM m); void pressButton(VM m); } class IdleState implements VendingState { void insertCoin(VM m) { m.setState(new HasMoneyState()); } void pressButton(VM m) { System.out.println("Insert coin first"); } } class HasMoneyState implements VendingState { void pressButton(VM m) { System.out.println("Dispensing..."); m.setState(new IdleState()); } }
IteratorBehavioral
Intent
Traverse a collection one by one WITHOUT knowing how it's stored internally.
When to use
Students in array, tree, or custom structure — all traversed with the same hasNext() / next().
Implementation
interface Iterator<T> { boolean hasNext(); T next(); } // Each collection provides its own iterator() // Usage — same code for any collection: Iterator<String> it = studentCollection.iterator(); while (it.hasNext()) System.out.println(it.next());
Template MethodBehavioral
Intent
Parent defines the fixed ORDER of steps. Subclasses override only the steps that differ.
When to use
Reports: fetchData → calculateMetrics → format → save. Same order, only calculate() differs per report.
Implementation
abstract class ReportGenerator { final void generateReport() { // template method — fixed order fetchData(); calculateMetrics(); // abstract — subclass overrides this formatOutput(); saveFile(); } protected abstract void calculateMetrics(); } class SalesReport extends ReportGenerator { void calculateMetrics() { System.out.println("Calculating sales..."); } }
MementoBehavioral
Intent
Save a snapshot of an object's state and restore it later. Without exposing internals.
When to use
Text editor undo · Game save/load checkpoint.
Implementation
// 3 roles: // Originator → creates Memento (snapshot) and restores from it // Memento → stores the saved state (just data, no logic) // Caretaker → Stack<Memento> history — push to save, pop to undo editor.type("Hello"); history.push(editor.save()); editor.type(" World"); // now "Hello World" editor.restore(history.pop()); // back to "Hello"
MediatorBehavioral
Intent
Objects don't talk to each other directly — all communication goes through a central mediator.
When to use
Chat room: users don't know each other. All messages go through ChatRoom.sendMessage().
Implementation
class ChatRoom implements ChatMediator { List<User> users; void sendMessage(String msg, User sender) { for (User u : users) if (u != sender) u.receive(sender.name + ": " + msg); } } // Users only call mediator.sendMessage() — never call each other
VisitorBehavioral
Intent
Add a new operation to existing objects WITHOUT modifying their classes.
When to use
Document (Paragraph, Image, Table) — add HTML export as a Visitor without touching each element class.
Implementation
interface DocumentVisitor { void visit(Paragraph p); void visit(Image i); } class HtmlExporter implements DocumentVisitor { void visit(Paragraph p) { System.out.println("<p>" + p.text + "</p>"); } void visit(Image i) { System.out.println("<img src='" + i.src + "'/>"); } } // Each element: accept(visitor) → visitor.visit(this)
InterpreterBehavioral
Intent
Define a grammar and interpret sentences in that language.
When to use
Math parser: "10 + 3 - 2" · SQL parser · Regex engine.
Implementation
interface Expression { int interpret(); } class NumberExpr implements Expression { int n; int interpret(){ return n; } } class AddExpr implements Expression { Expression left, right; int interpret() { return left.interpret() + right.interpret(); } } // "10 + 3" = new AddExpr(new NumberExpr(10), new NumberExpr(3)).interpret() = 13
Code smells — what, examples, and fixes
What is a code smell?
A warning sign that your code has a bad design choice — not a bug, but a signal that problems will come later.
1. Switch statement / Long if-else on types
Checking type/payment/format with if-else or switch. Every new type forces you to edit this method.
if (type.equals("CreditCard")) { ... } else if (type.equals("PayPal")) { ... } else if (type.equals("Cash")) { ... } // adding "Bitcoin"? edit this method!
Fix: Strategy Pattern — each type becomes its own class. New type = new class only. Existing code untouched.
2. Tight coupling (especially through inheritance)
Classes depend directly on concrete classes instead of interfaces. Changing one breaks others.
class OrderProcessor { CreditCardPayment payment = new CreditCardPayment(); // tightly coupled! }
Fix: Program to interfaces — use PaymentStrategy interface. Inject the implementation. Loose coupling.
3. Long method / Long class
One method or class doing too many things. Hard to read, test, and maintain.
void processOrder() { // validate ... 20 lines // calculate total ... 15 lines // process payment ... 20 lines // send email ... 15 lines ← this is one giant mess }
Fix: Extract methods and apply Single Responsibility Principle (SRP) — one class = one job. Split into validateOrder(), calculateTotal(), processPayment(), sendEmail().
4. Poor abstraction (class doing unrelated things)
A class or method handles things that don't belong together.
class User { void login() { ... } void sendEmail() { ... } // email logic doesn't belong in User! void generateReport() { ... } }
Fix: SRP + Facade / separate classes — User handles user logic. EmailService handles email. ReportService handles reports.
5. Duplicate code
Same logic copy-pasted in multiple places. Fix one → must fix all. Easy to miss one.
// SalesReport class: void fetchData() { connect(); query("sales"); } // InventoryReport class: void fetchData() { connect(); query("inventory"); } // copy-paste!
Fix: Template Method Pattern — put the common steps in a parent class. Only override what differs.
6. Lack of separation of concerns
Business logic mixed with UI or database code in the same place.
class ProductView { void render() { // fetch from DB... 10 lines // business calculations... 10 lines // display HTML... 10 lines ← all 3 concerns in one place! } }
Fix: MVC Pattern — Model handles data, View handles display, Controller handles logic. Each has one concern.
MVC and combined design patterns
M
Model — data and business logic
Classes for each DB table. Knows API endpoints. Manages application data and rules. Has NO idea how anything looks on screen.
V
View — the user interface (UI)
What the user sees. Displays data from the Model. Sends user actions (clicks, inputs) to the Controller. Contains NO business logic.
C
Controller — the middleman
Sits between Model and View. Receives user input, validates it, updates the Model, then tells View to refresh. Logic lives here.
Design patterns used inside MVC
Observer
Model (Subject) notifies View (Observer) on every state change. View auto-updates.
Strategy
Controller uses Strategy to handle different user actions — interchangeable and extendable.
Singleton
One shared Model or Controller instance across the whole app (e.g. one DB connection).
Composite
View: UI components nested inside each other — buttons in panels in windows.
Factory Method
Controller creates View or Model objects without depending on concrete types.
MCQ answers — memorize these:
Who handles user input? → Controller
Model ↔ View relationship uses which pattern? → Observer
Controller handles different actions using? → Strategy
Model contains? → Business logic + application data
Which statement is TRUE about MVC? → Controller mediates between View and Model
Can View directly modify the Model? → No
