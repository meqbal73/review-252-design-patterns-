What are design patterns? ما هي أنماط التصميم؟
Proven, reusable solutions to commonly occurring problems in software design. Not copy-paste code — they are ideas / templates you apply to your situation. حلول مجربة وقابلة لإعادة الاستخدام لمشاكل متكررة في تصميم البرمجيات. ليست كودًا تنسخه — بل هي أفكار وقوالب تطبّقها على حالتك.
Solve complex problemsحل المشاكل المعقدة
Give you a tested solution instead of inventing one from scratch.تمنحك حلًا مجربًا بدلًا من اختراع الحل من الصفر.
Improve readabilityتحسين قراءة الكود
Other developers recognize the pattern and understand your code faster.المطورون الآخرون يتعرفون على النمط ويفهمون كودك بسرعة.
Loose couplingارتباط خفيف
Add new features without breaking existing code. (Open/Closed Principle)أضف ميزات جديدة دون كسر الكود الموجود. (مبدأ Open/Closed)
Core OOP design principles (exam favourite!)مبادئ التصميم الأساسية (مهمة جداً في الاختبار!)
1. Program to interfaces١. البرمجة للواجهات
Write abstract classes/interfaces first. Client code uses the interface type, NOT the concrete class type.اكتب الواجهات أولًا. الكود العميل يستخدم نوع الواجهة، وليس الكلاس الفعلي.
2. Single Responsibility (SRP)٢. مبدأ المسؤولية الواحدة
A class should have ONE job/responsibility only. If it does two things, split it.الكلاس يجب أن يكون له وظيفة واحدة فقط. إذا كان يفعل شيئين، اقسمه.
3. Open/Closed Principle٣. مبدأ Open/Closed
Classes should be open for extension but closed for modification. Add new code, don't change old code.الكلاسات يجب أن تكون مفتوحة للتوسع ومغلقة للتعديل. أضف كودًا جديدًا ولا تغير القديم.

Creational patternsالأنماط الإنشائية
SingletonCreational
Intentالهدف
Only ONE instance exists in the whole app. Everyone shares it.نسخة واحدة فقط من الكلاس في كامل التطبيق. الجميع يشاركها.
When to useمتى تستخدمه
Database connection · App logger · Config settingsاتصال قاعدة البيانات · مسجّل التطبيق · إعدادات التكوين
OOP Principleمبدأ OOP SRP — the singleton class manages only one shared resource. The private constructor enforces this contract. SRP — الكلاس يدير مورداً مشتركاً واحداً فقط. المُنشئ الخاص يُطبّق هذا العقد.
Implementationالتطبيق
Java
class Database {
    private static Database instance;

    private Database() { }  // no one can do: new Database()

    public static Database getInstance() {
        if (instance == null)
            instance = new Database();
        return instance;
    }
    public void query(String sql) { /* run query */ }
}

// Usage
Database db1 = Database.getInstance();
Database db2 = Database.getInstance();
System.out.println(db1 == db2); // true — same object
PrototypeCreational
Intentالهدف
Clone/copy an existing object instead of building a new one from scratch.استنساخ كائن موجود بدلاً من بناء كائن جديد من الصفر.
When to useمتى تستخدمه
Game character duplication · Any expensive-to-create objectنسخ شخصية في لعبة · أي كائن تكلفة إنشائه عالية
OOP Principleمبدأ OOP Program to interfaces — client calls clone() on the Prototype interface without knowing the concrete class. البرمجة للواجهات — الكود العميل يستدعي clone() على الواجهة دون معرفة الكلاس الفعلي.
Implementationالتطبيق
Java
class GameCharacter implements Cloneable {
    String name;
    int health;
    String weapon;

    public GameCharacter(String name, int health, String weapon) {
        this.name = name; this.health = health; this.weapon = weapon;
    }
    public GameCharacter clone() {
        try { return (GameCharacter) super.clone(); }
        catch (CloneNotSupportedException e) { return null; }
    }
}

// Usage
GameCharacter c1 = new GameCharacter("Ali", 100, "Sword");
GameCharacter c2 = c1.clone();  // cheap copy — no new construction
c2.name = "Ali2";               // c1.name still "Ali"
BuilderCreational
Intentالهدف
Build complex objects step by step. Same construction process, different results.بناء كائنات معقدة خطوة بخطوة. نفس عملية البناء، نتائج مختلفة.
When to useمتى تستخدمه
Burger with optional items · HTTP request with optional fieldsبرغر بمكونات اختيارية · طلب HTTP بحقول اختيارية
OOP Principleمبدأ OOP SRP — separates the construction logic from the object itself. Open/Closed — add new optional fields without touching existing build methods. SRP — يفصل منطق البناء عن الكائن نفسه. Open/Closed — أضف حقولاً اختيارية جديدة دون تعديل الكود الموجود.
Implementationالتطبيق
Java
class Burger {
    String bun, patty, sauce;
    boolean cheese, lettuce;

    static class Builder {
        private Burger burger = new Burger();

        public Builder withBun(String b)    { burger.bun = b;    return this; }
        public Builder withPatty(String p)  { burger.patty = p;  return this; }
        public Builder withSauce(String s)  { burger.sauce = s;  return this; }
        public Builder withCheese()          { burger.cheese = true; return this; }
        public Burger  build()               { return burger; }
    }
}

// Usage — fluent chain
Burger b = new Burger.Builder()
    .withBun("sesame")
    .withPatty("beef")
    .withCheese()
    .withSauce("bbq")
    .build();
Factory MethodCreational
Intentالهدف
Parent defines "create a transport" — subclass decides Truck or Ship. Client never calls new directly.الأب يحدد "أنشئ وسيلة نقل" — الابن يقرر Truck أو Ship. الكود العميل لا يستخدم new مباشرة.
When to useمتى تستخدمه
When you don't know which type to create. Easy to add new types later.عندما لا تعرف مسبقاً أي نوع ستنشئ. سهولة إضافة أنواع جديدة.
OOP Principleمبدأ OOP Open/Closed — add a new transport type by adding a new subclass only. Program to interfaces — client uses Transport interface, not Truck or Ship. Open/Closed — أضف نوع نقل جديد بإضافة كلاس فرعي فقط. البرمجة للواجهات — الكود العميل يستخدم واجهة Transport.
Implementationالتطبيق
Java
interface Transport { void deliver(); }

class Truck implements Transport {
    public void deliver() { System.out.println("By road"); }
}
class Ship implements Transport {
    public void deliver() { System.out.println("By sea"); }
}

abstract class Logistics {
    abstract Transport createTransport(); // factory method
    void planDelivery() { createTransport().deliver(); }
}
class RoadLogistics extends Logistics {
    Transport createTransport() { return new Truck(); }
}
class SeaLogistics extends Logistics {
    Transport createTransport() { return new Ship(); }
}
Abstract FactoryCreational
Intentالهدف
Produce a whole FAMILY of related objects. Swap the factory = swap everything at once.إنتاج عائلة كاملة من الكائنات المترابطة. استبدال المصنع = استبدال كل شيء دفعة واحدة.
When to useمتى تستخدمه
UI toolkit: MacFactory → Mac button + checkbox. WindowsFactory → Windows button + checkbox.مجموعة واجهة: MacFactory تنتج زر وصندوق تحقق بأسلوب Mac. WindowsFactory بأسلوب Windows.
OOP Principleمبدأ OOP Program to interfaces — client uses UIFactory interface. Open/Closed — new platform = new factory class only. البرمجة للواجهات — الكود العميل يستخدم UIFactory. Open/Closed — منصة جديدة = كلاس مصنع جديد فقط.
Implementationالتطبيق
Java
interface UIFactory {
    Button   createButton();
    Checkbox createCheckbox();
}
class MacFactory implements UIFactory {
    public Button   createButton()   { return new MacButton(); }
    public Checkbox createCheckbox() { return new MacCheckbox(); }
}
class WindowsFactory implements UIFactory {
    public Button   createButton()   { return new WinButton(); }
    public Checkbox createCheckbox() { return new WinCheckbox(); }
}

// Swap MacFactory ↔ WindowsFactory = swap entire UI kit
UIFactory factory = new MacFactory();
factory.createButton().render();  // Mac Button
Structural patternsالأنماط الهيكلية
AdapterStructural
Intentالهدف
Make two incompatible interfaces work together via a wrapper. Like a power plug adapter.جعل واجهتين غير متوافقتين تعملان معاً عبر مُغلِّف. مثل محول التيار الكهربائي.
When to useمتى تستخدمه
Client sends city name. Library expects GPS coordinates. Adapter converts city → lat/lon → delegates.الكود العميل يرسل اسم مدينة. المكتبة تتوقع إحداثيات GPS. المحوّل يحوّل الاسم → إحداثيات → يفوّض.
OOP Principleمبدأ OOP Program to interfaces — client only knows FajrTimeCity interface, never FajrTimeGeo. Open/Closed — existing FajrTimeGeo is untouched, we wrap it. البرمجة للواجهات — الكود العميل يعرف فقط FajrTimeCity. Open/Closed — لا نعدّل FajrTimeGeo، نلفّه فقط.
Implementationالتطبيق
Java
// Target interface (what client expects)
interface FajrTimeCity {
    String getFajrTime(String cityName);
}
// Adaptee (existing, incompatible)
class FajrTimeGeo {
    public String getFajrTime(double lat, double lon) {
        return "Fajr time is 5:39 AM";
    }
}
// Adapter — bridges the two
class FajrTimeAdapter implements FajrTimeCity {
    private FajrTimeGeo geo;

    public FajrTimeAdapter(FajrTimeGeo geo) { this.geo = geo; }

    public String getFajrTime(String city) {
        double lat = getLatitude(city);
        double lon = getLongitude(city);
        return geo.getFajrTime(lat, lon); // delegate
    }
    private double getLatitude(String c)  { return 21.5433; }
    private double getLongitude(String c) { return 39.1728; }
}
BridgeStructural
Intentالهدف
Separate "what it is" (abstraction) from "how it works" (implementation) so both can vary independently.فصل "ما هو" (التجريد) عن "كيف يعمل" (التطبيق) حتى يتغير كل منهما باستقلالية.
OOP Principleمبدأ OOP Program to interfaces — Shape holds a Color interface reference, not a concrete Red or Blue. Avoids class explosion. البرمجة للواجهات — Shape تحمل مرجع لواجهة Color لا لكلاس محدد. تتجنب انفجار عدد الكلاسات.
Implementationالتطبيق
Java
interface Color { String fill(); }
class Red  implements Color { public String fill() { return "red"; } }
class Blue implements Color { public String fill() { return "blue"; } }

abstract class Shape {
    protected Color color; // bridge to implementation
    public Shape(Color c) { this.color = c; }
    abstract void draw();
}
class Circle extends Shape {
    public Circle(Color c) { super(c); }
    public void draw() { System.out.println("Circle in " + color.fill()); }
}

// 2 shapes × 2 colors = 4 combos, only 4 classes (not 4 subclasses!)
new Circle(new Red()).draw();   // Circle in red
new Circle(new Blue()).draw();  // Circle in blue
CompositeStructural
Intentالهدف
Treat a single object and a group of objects the SAME way. Tree structure (part-whole).معاملة الكائن الواحد والمجموعة بنفس الطريقة. هيكل شجرة (جزء-كل).
OOP Principleمبدأ OOP Program to interfaces — both File and Folder implement FileSystemItem. Client never checks "is this a file or folder?" البرمجة للواجهات — كلٌّ من File و Folder تطبّق FileSystemItem. الكود العميل لا يتحقق "هل هذا ملف أم مجلد؟"
Implementationالتطبيق
Java
interface FileSystemItem { int getSize(); }

class File implements FileSystemItem {
    int size;
    public int getSize() { return size; }
}
class Folder implements FileSystemItem {
    List<FileSystemItem> children = new ArrayList<>();
    public void add(FileSystemItem i) { children.add(i); }
    public int getSize() {
        return children.stream().mapToInt(FileSystemItem::getSize).sum();
    }
}

// Same call works on both File and Folder
Folder root = new Folder();
root.add(new File(5));  root.add(new File(20));
System.out.println(root.getSize()); // 25
DecoratorStructural
Intentالهدف
Add new behavior by WRAPPING an object — without changing its class. Stack layers dynamically.إضافة سلوك جديد بلفّ الكائن — دون تغيير كلاسه. تراكم الطبقات بشكل ديناميكي.
OOP Principleمبدأ OOP Open/Closed — add behaviors without modifying Coffee class. Program to interfaces — every decorator implements Coffee. Open/Closed — أضف سلوكاً دون تعديل كلاس Coffee. البرمجة للواجهات — كل محسّن يطبّق واجهة Coffee.
Implementationالتطبيق
Java
interface Coffee { String getDesc(); double getCost(); }

class SimpleCoffee implements Coffee {
    public String getDesc() { return "Coffee"; }
    public double getCost() { return 1.0; }
}
abstract class CoffeeDecorator implements Coffee {
    protected Coffee coffee;
    public CoffeeDecorator(Coffee c) { this.coffee = c; }
}
class MilkDecorator extends CoffeeDecorator {
    public MilkDecorator(Coffee c) { super(c); }
    public String getDesc() { return coffee.getDesc() + ", Milk"; }
    public double getCost() { return coffee.getCost() + 0.5; }
}
class SugarDecorator extends CoffeeDecorator {
    public SugarDecorator(Coffee c) { super(c); }
    public String getDesc() { return coffee.getDesc() + ", Sugar"; }
    public double getCost() { return coffee.getCost() + 0.25; }
}

// Stack decorators
Coffee c = new SugarDecorator(new MilkDecorator(new SimpleCoffee()));
System.out.println(c.getDesc()); // Coffee, Milk, Sugar
System.out.println(c.getCost()); // 1.75
FacadeStructural
Intentالهدف
One simple interface hides a complex subsystem. Client calls ONE method instead of many.واجهة بسيطة واحدة تخفي نظاماً فرعياً معقداً. الكود العميل يستدعي طريقة واحدة بدلاً من كثير.
OOP Principleمبدأ OOP SRP — each subsystem class has its own job. The Facade just coordinates them. Reduces coupling between client and subsystem. SRP — كل كلاس في النظام الفرعي له وظيفة واحدة. الـ Facade يُنسّق فقط. يقلل الارتباط بين العميل والنظام.
Implementationالتطبيق
Java
class TV          { void on() { System.out.println("TV on"); } }
class SoundSystem { void on() { System.out.println("Sound on"); }
                     void setVolume(int v) {} }
class Lights      { void dim(int pct) { System.out.println("Lights at "+pct+"%"); } }

// Facade — one simple method
class HomeTheaterFacade {
    private TV tv = new TV();
    private SoundSystem sound = new SoundSystem();
    private Lights lights = new Lights();

    public void watchMovie() {
        lights.dim(20);
        tv.on();
        sound.on(); sound.setVolume(30);
        System.out.println("Enjoy!");
    }
}

// Client only needs ONE call
new HomeTheaterFacade().watchMovie();
FlyweightStructural
Intentالهدف
Save memory by SHARING intrinsic (common) state. Extrinsic (unique) state is passed in at runtime.توفير الذاكرة بمشاركة الحالة الداخلية المشتركة. الحالة الخارجية الفريدة تُمرَّر عند التشغيل.
OOP Principleمبدأ OOP SRP — TreeType manages shared state only. Tree manages unique position only. Clean separation of concerns. SRP — TreeType تدير الحالة المشتركة فقط. Tree تدير الموقع الفريد فقط. فصل واضح للمسؤوليات.
Implementationالتطبيق
Java
// Flyweight — shared (intrinsic) state
class TreeType {
    String name, texture;
    void draw(int x, int y) {
        System.out.println("Drawing " + name + " at ("+x+","+y+")");
    }
}
// Factory — returns cached flyweights
class TreeFactory {
    static Map<String,TreeType> cache = new HashMap<>();
    static TreeType get(String name, String tex) {
        return cache.computeIfAbsent(name, k -> new TreeType());
    }
}
// Context — unique (extrinsic) state only
class Tree {
    int x, y; TreeType type;
    void draw() { type.draw(x, y); }
}
// 1,000,000 Trees — only 2 TreeType objects in memory!
ProxyStructural
Intentالهدف
A stand-in that controls access to the real object. Same interface. Adds validation / lazy-load / caching.بديل يتحكم في الوصول للكائن الحقيقي. نفس الواجهة. يضيف تحقق / تحميل متأخر / تخزين مؤقت.
OOP Principleمبدأ OOP Program to interfaces — client uses SpeechToTextService, never cares if it's Proxy or Real. Open/Closed — adding caching = new proxy class, no change to real service. البرمجة للواجهات — العميل يستخدم SpeechToTextService دون اهتمام هل هو Proxy أم حقيقي. Open/Closed — إضافة التخزين المؤقت = كلاس Proxy جديد فقط.
Implementationالتطبيق
Java
interface SpeechToTextService {
    String convertSpeechToText(String audioFile);
}
class RealSpeechToText implements SpeechToTextService {
    public RealSpeechToText() { System.out.println("Loading model..."); }
    public String convertSpeechToText(String f) { return "Transcription"; }
}
// Protection Proxy — validates before delegating
class SpeechToTextProxy implements SpeechToTextService {
    private RealSpeechToText real; // lazy — not created yet
    private String apiKey;

    public SpeechToTextProxy(String key) { this.apiKey = key; }

    private boolean isValid() { return apiKey != null && !apiKey.isEmpty(); }

    public String convertSpeechToText(String file) {
        if (!isValid()) return "Access denied";
        if (real == null) real = new RealSpeechToText(); // lazy init
        return real.convertSpeechToText(file);
    }
}
Behavioral patternsالأنماط السلوكية
Chain of ResponsibilityBehavioral
Intentالهدف
Pass a request along a chain. Each handler processes it OR passes it to the next one.تمرير الطلب على سلسلة. كل معالج إما يعالجه أو يمرره للتالي.
OOP Principleمبدأ OOP Open/Closed — add a new level by adding a new handler class. SRP — each handler has one decision: handle or forward. Open/Closed — أضف مستوى جديداً بإضافة كلاس معالج جديد. SRP — كل معالج له قرار واحد: يعالج أو يمرر.
Implementationالتطبيق
Java
abstract class SupportHandler {
    protected SupportHandler next;
    public void setNext(SupportHandler h) { next = h; }
    public abstract void handle(int level);
}
class Level1 extends SupportHandler {
    public void handle(int level) {
        if (level == 1) System.out.println("Level1 handled");
        else if (next != null) next.handle(level); // pass forward
    }
}
class Level2 extends SupportHandler {
    public void handle(int level) {
        if (level == 2) System.out.println("Level2 handled");
        else if (next != null) next.handle(level);
    }
}
// Build chain: L1 → L2 → L3
Level1 l1 = new Level1(); Level2 l2 = new Level2();
l1.setNext(l2);
l1.handle(2); // Level2 handled
CommandBehavioral
Intentالهدف
Wrap a request as an object — enables undo, queue, logging.لفّ الطلب ككائن — يُمكّن التراجع والطابور والتسجيل.
OOP Principleمبدأ OOP SRP — Command encapsulates ONE action. Open/Closed — new action = new Command class, no change to invoker/editor. SRP — Command يغلّف إجراءً واحداً. Open/Closed — إجراء جديد = كلاس Command جديد فقط.
Implementationالتطبيق
Java
interface Command { void execute(); void undo(); }

class WriteCommand implements Command {
    private TextEditor editor; private String text;
    public WriteCommand(TextEditor e, String t) { editor=e; text=t; }
    public void execute() { editor.write(text); }
    public void undo()    { editor.deleteLast(text.length()); }
}

// Invoker — holds undo history
class EditorInvoker {
    private Stack<Command> history = new Stack<>();
    public void run(Command c)  { c.execute(); history.push(c); }
    public void undo() { if (!history.isEmpty()) history.pop().undo(); }
}

// Usage
TextEditor editor = new TextEditor();
EditorInvoker inv = new EditorInvoker();
inv.run(new WriteCommand(editor, "Hello"));
inv.run(new WriteCommand(editor, " World"));
inv.undo(); // removes " World"
ObserverBehavioral
Intentالهدف
One subject notifies ALL observers automatically when it changes state. One-to-many relationship.الموضوع يُخطر جميع المراقبين تلقائياً عند تغيير حالته. علاقة واحد-لكثير.
OOP Principleمبدأ OOP Program to interfaces — Subject uses Observer interface, never knows concrete Fan. Open/Closed — add new observer type without changing Player. البرمجة للواجهات — Subject يستخدم واجهة Observer، لا يعرف Fan المحدد. Open/Closed — أضف نوع مراقب جديد دون تغيير Player.
Implementationالتطبيق
Java
interface Observer { void update(Player p); }

class Player implements Subject {
    private String name;
    private int currentPoints;
    private double sessionAvg;
    private List<Observer> observers = new ArrayList<>();

    public void attach(Observer o) { observers.add(o); }
    public void detach(Observer o) { observers.remove(o); }
    public void notifyObservers() {
        for (Observer o : observers) o.update(this);
    }
    public void setCurrentPoints(int pts) {
        this.currentPoints = pts;
        if (currentPoints > sessionAvg) notifyObservers();
    }
    public String getName()       { return name; }
    public int    getCurrentPoints() { return currentPoints; }
}
class Fan implements Observer {
    public void update(Player p) {
        System.out.println("Player "+p.getName()+" exceeded avg with "+p.getCurrentPoints());
    }
}
StrategyBehavioral
Intentالهدف
Family of interchangeable algorithms. Pick one at runtime. Eliminates if/else chains.عائلة من الخوارزميات القابلة للتبادل. اختر واحدة أثناء التشغيل. يُلغي سلاسل if/else.
OOP Principleمبدأ OOP Open/Closed — add new payment type = new class only, never touch OrderProcessor. Program to interfaces — OrderProcessor holds PaymentStrategy, not CreditCard. Open/Closed — نوع دفع جديد = كلاس جديد فقط، لا تعديل على OrderProcessor. البرمجة للواجهات — يحمل PaymentStrategy لا CreditCard.
Implementationالتطبيق
Java
interface PaymentStrategy { void pay(double amount); }

class CreditCardPayment implements PaymentStrategy {
    public void pay(double a) { System.out.println("Paid $"+a+" by card"); }
}
class PayPalPayment implements PaymentStrategy {
    public void pay(double a) { System.out.println("Paid $"+a+" via PayPal"); }
}
class OrderProcessor {
    private PaymentStrategy strategy;
    public void setStrategy(PaymentStrategy s) { strategy = s; }
    public void processOrder(double total) {
        strategy.pay(total); // no if/else!
    }
}

// Usage — swap strategy at runtime
OrderProcessor p = new OrderProcessor();
p.setStrategy(new CreditCardPayment()); p.processOrder(99.99);
p.setStrategy(new PayPalPayment());    p.processOrder(49.99);
StateBehavioral
Intentالهدف
Object changes behavior when its internal state changes — same method, different result per state.الكائن يغير سلوكه عند تغير حالته الداخلية — نفس الطريقة، نتيجة مختلفة لكل حالة.
OOP Principleمبدأ OOP SRP — each state handles its own behavior. Open/Closed — new state = new class only. SRP — كل حالة تتولى سلوكها الخاص. Open/Closed — حالة جديدة = كلاس جديد فقط.
Implementationالتطبيق
Java
interface VendingState {
    void insertCoin(VendingMachine m);
    void pressButton(VendingMachine m);
}
class IdleState implements VendingState {
    public void insertCoin(VendingMachine m) { m.setState(new HasMoneyState()); }
    public void pressButton(VendingMachine m) { System.out.println("Insert coin first"); }
}
class HasMoneyState implements VendingState {
    public void insertCoin(VendingMachine m) { System.out.println("Coin already in"); }
    public void pressButton(VendingMachine m) {
        System.out.println("Dispensing..."); m.setState(new IdleState());
    }
}
class VendingMachine {
    private VendingState state = new IdleState();
    public void setState(VendingState s) { state = s; }
    public void insertCoin()   { state.insertCoin(this); }
    public void pressButton() { state.pressButton(this); }
}
IteratorBehavioral
Intentالهدف
Traverse any collection sequentially without exposing its internal structure.تصفح أي مجموعة بشكل تسلسلي دون كشف بنيتها الداخلية.
OOP Principleمبدأ OOP Program to interfaces — client uses Iterator interface, never cares if collection is array, tree, or custom. SRP — collection manages data; Iterator manages traversal. البرمجة للواجهات — العميل يستخدم Iterator بغض النظر عن نوع المجموعة. SRP — المجموعة تدير البيانات؛ Iterator يدير التصفح.
Implementationالتطبيق
Java
interface Iterator<T> { boolean hasNext(); T next(); }

class StudentArray {
    private String[] students; private int size = 0;
    public void add(String s) { students[size++] = s; }

    public Iterator<String> iterator() {
        return new Iterator<String>() {
            int i = 0;
            public boolean hasNext() { return i < size; }
            public String  next()    { return students[i++]; }
        };
    }
}
// Same traversal code works for ANY collection type
Iterator<String> it = collection.iterator();
while (it.hasNext()) System.out.println(it.next());
Template MethodBehavioral
Intentالهدف
Parent defines the fixed ORDER of steps. Subclasses override ONLY the steps that differ.الأب يحدد الترتيب الثابت للخطوات. الأبناء يتجاوزون الخطوات المختلفة فقط.
OOP Principleمبدأ OOP Open/Closed — new report type = new subclass only. SRP — parent owns the algorithm structure; subclass owns the specific step logic. Open/Closed — نوع تقرير جديد = كلاس فرعي جديد. SRP — الأب يملك هيكل الخوارزمية؛ الابن يملك منطق الخطوة المحددة.
Implementationالتطبيق
Java
abstract class ReportGenerator {
    // Template method — fixed order, cannot be overridden
    public final void generateReport() {
        fetchData();
        calculateMetrics(); // ← only this differs per report
        formatOutput();
        saveFile();
    }
    protected void fetchData()    { System.out.println("Fetching data..."); }
    protected abstract void calculateMetrics(); // subclass implements
    protected void formatOutput() { System.out.println("Formatting..."); }
    protected void saveFile()     { System.out.println("Saving..."); }
}
class SalesReport extends ReportGenerator {
    protected void calculateMetrics() { System.out.println("Calculating sales"); }
}
class InventoryReport extends ReportGenerator {
    protected void calculateMetrics() { System.out.println("Calculating stock"); }
}
MementoBehavioral
Intentالهدف
Save and restore object state without exposing its internals. Three roles: Originator, Memento, Caretaker.حفظ واستعادة حالة الكائن دون كشف تفاصيله. ثلاثة أدوار: المنشئ، الذاكرة، الحارس.
OOP Principleمبدأ OOP SRP — Originator manages state, Memento stores snapshot, Caretaker manages history. Each has one job. SRP — المنشئ يدير الحالة، Memento يخزن اللقطة، Caretaker يدير السجل. كل واحد له وظيفة واحدة.
Implementationالتطبيق
Java
class EditorMemento { // stores snapshot
    final String text;
    EditorMemento(String t) { text = t; }
}
class Editor { // Originator
    private String text = "";
    public void type(String s) { text += s; }
    public EditorMemento save()                { return new EditorMemento(text); }
    public void           restore(EditorMemento m) { text = m.text; }
    public String         getText()              { return text; }
}
class History { // Caretaker
    private Stack<EditorMemento> stack = new Stack<>();
    public void push(EditorMemento m) { stack.push(m); }
    public EditorMemento pop()       { return stack.pop(); }
}
Editor e = new Editor(); History h = new History();
e.type("Hello"); h.push(e.save());
e.type(" World"); // now "Hello World"
e.restore(h.pop()); // back to "Hello"
MediatorBehavioral
Intentالهدف
Objects don't talk directly — all communication goes through a central mediator.الكائنات لا تتحدث مباشرة — كل التواصل يمر عبر وسيط مركزي.
OOP Principleمبدأ OOP SRP — each user only sends/receives messages. The mediator manages routing. Reduces chaotic coupling. SRP — كل مستخدم يرسل/يستقبل فقط. الوسيط يدير التوجيه. يقلل الارتباط الفوضوي.
Implementationالتطبيق
Java
class ChatRoom { // Mediator
    private List<User> users = new ArrayList<>();
    void addUser(User u) { users.add(u); }
    void sendMessage(String msg, User sender) {
        for (User u : users)
            if (u != sender) u.receive(sender.name + ": " + msg);
    }
}
class User {
    String name; private ChatRoom room;
    void send(String msg)    { room.sendMessage(msg, this); }
    void receive(String msg) { System.out.println(name+" got: "+msg); }
}
// Users never reference each other directly
ChatRoom room = new ChatRoom();
User ali = new User(); ali.name="Ali"; ali.room=room;
room.addUser(ali); ali.send("Hello!");
VisitorBehavioral
Intentالهدف
Add new operations to objects WITHOUT modifying their classes. New operation = new Visitor class.إضافة عمليات جديدة للكائنات دون تعديل كلاساتها. عملية جديدة = كلاس Visitor جديد.
OOP Principleمبدأ OOP Open/Closed — add HTML export without touching Paragraph or Image. SRP — elements handle structure; visitors handle operations. Open/Closed — أضف تصدير HTML دون تعديل Paragraph أو Image. SRP — العناصر تدير الهيكل؛ الزوار يديرون العمليات.
Implementationالتطبيق
Java
interface DocumentVisitor {
    void visit(Paragraph p); void visit(Image i);
}
class Paragraph { String text; void accept(DocumentVisitor v) { v.visit(this); } }
class Image     { String src;  void accept(DocumentVisitor v) { v.visit(this); } }

// New operation: HTML export (no change to Paragraph or Image!)
class HtmlExporter implements DocumentVisitor {
    public void visit(Paragraph p) { System.out.println("<p>"+p.text+"</p>"); }
    public void visit(Image i)     { System.out.println("<img src='"+i.src+"'/>"); }
}
// Usage
DocumentVisitor exporter = new HtmlExporter();
elements.forEach(el -> el.accept(exporter));
InterpreterBehavioral
Intentالهدف
Define a grammar and interpret sentences in that language.تحديد قواعد لغة وتفسير جملها.
OOP Principleمبدأ OOP Program to interfaces — all expressions implement the same interpret() interface. Open/Closed — new grammar rule = new Expression class. البرمجة للواجهات — جميع التعبيرات تطبّق واجهة interpret(). Open/Closed — قاعدة جديدة = كلاس Expression جديد.
Implementationالتطبيق
Java
interface Expression { int interpret(); }

class NumberExpr implements Expression {
    int n; NumberExpr(int n) { this.n = n; }
    public int interpret() { return n; }
}
class AddExpr implements Expression {
    Expression left, right;
    public int interpret() { return left.interpret() + right.interpret(); }
}
class SubtractExpr implements Expression {
    Expression left, right;
    public int interpret() { return left.interpret() - right.interpret(); }
}

// Interpret "10 + 3 - 2"
Expression expr = new SubtractExpr() {{
    left = new AddExpr() {{ left=new NumberExpr(10); right=new NumberExpr(3); }};
    right = new NumberExpr(2);
}};
System.out.println(expr.interpret()); // 11
Code smells — signals of bad designروائح الكود — إشارات التصميم السيئ
A code smell is NOT a bug — it is a warning sign that your design has a problem that will cause trouble later. رائحة الكود ليست خطأً برمجياً — بل إشارة تحذير أن تصميمك فيه مشكلة ستسبب متاعب لاحقاً.
1. Switch statement / Long if-else on types١. جملة switch / سلسلة if-else طويلة على الأنواع
Every new type forces you to edit this method. Violates Open/Closed Principle.كل نوع جديد يجبرك على تعديل هذه الطريقة. يخالف مبدأ Open/Closed.
smell
if (type.equals("CreditCard")) { ... }
else if (type.equals("PayPal")) { ... }
else if (type.equals("Cash"))   { ... }
// Adding "Bitcoin"? You must edit this method!
Fix: Strategy Pattern — each type becomes its own class. New type = add a class, never edit existing code.الحل: نمط Strategy — كل نوع يصبح كلاسه الخاص. نوع جديد = أضف كلاساً ولا تعدّل الكود الموجود.
2. Tight coupling (depending on concrete classes)٢. الارتباط الشديد (الاعتماد على الكلاسات المحددة)
Classes depend directly on concrete classes instead of interfaces. Changing one breaks others.الكلاسات تعتمد على كلاسات محددة بدلاً من الواجهات. تغيير واحد يكسر الباقي.
smell
class OrderProcessor {
    CreditCardPayment payment = new CreditCardPayment(); // tightly coupled!
}
Fix: Program to interfaces — use PaymentStrategy. Inject the concrete class from outside (Dependency Injection).الحل: برمجة للواجهات — استخدم PaymentStrategy. أدخل الكلاس المحدد من الخارج (Dependency Injection).
3. Long method / Long class٣. الطريقة الطويلة / الكلاس الطويل
One method or class doing too many things. Hard to read, test, and maintain.طريقة أو كلاس يفعل أشياء كثيرة جداً. صعب القراءة والاختبار والصيانة.
smell
void processOrder() {
    // validate ... 20 lines
    // calculate total ... 15 lines
    // process payment ... 20 lines
    // send email ... 15 lines  ← one giant mess
}
Fix: SRP — split into validateOrder(), calculateTotal(), processPayment(), sendEmail(). One method = one job.الحل: SRP — اقسم إلى validateOrder()، calculateTotal()، processPayment()، sendEmail(). طريقة واحدة = وظيفة واحدة.
4. Poor abstraction (class doing unrelated things)٤. التجريد السيئ (الكلاس يفعل أشياء غير مترابطة)
A class handles things that don't belong together. Violates SRP.الكلاس يتعامل مع أشياء لا تنتمي لبعضها. يخالف SRP.
smell
class User {
    void login() { ... }
    void sendEmail() { ... }     // email doesn't belong in User!
    void generateReport() { ... } // reports don't either!
}
Fix: SRP — User handles user logic only. Create EmailService, ReportService as separate classes.الحل: SRP — User يدير منطق المستخدم فقط. أنشئ EmailService و ReportService ككلاسات منفصلة.
5. Duplicate code٥. ازدواجية الكود
Same logic copy-pasted in multiple places. Fix one → must fix all. Easy to miss one.نفس المنطق مكرر في أماكن متعددة. إصلاح واحد يعني إصلاح الجميع. سهل أن تنسى واحداً.
smell
// In SalesReport:
void fetchData() { connect(); query("sales"); }
// In InventoryReport (copy-pasted!):
void fetchData() { connect(); query("inventory"); }
Fix: Template Method — put shared steps in parent class. Subclasses override only what differs.الحل: Template Method — ضع الخطوات المشتركة في الكلاس الأب. الأبناء يتجاوزون فقط ما يختلف.
6. Lack of separation of concerns٦. غياب فصل الاهتمامات
Business logic mixed with UI or database code in the same place.منطق العمل مخلوط مع كود الواجهة أو قاعدة البيانات في نفس المكان.
smell
class ProductView {
    void render() {
        // fetch from DB ... 10 lines
        // business calculations ... 10 lines
        // display HTML ... 10 lines  ← 3 concerns in one place!
    }
}
Fix: MVC Pattern — Model handles data, View handles display, Controller handles logic. Each has one concern.الحل: نمط MVC — Model يدير البيانات، View يدير العرض، Controller يدير المنطق. كل طرف له اهتمام واحد.
MVC and combined design patternsMVC والأنماط المدمجة
M
Model — data and business logicModel — البيانات ومنطق العمل
Classes for each DB table. Knows API endpoints. Manages data and rules. Has NO idea how anything looks on screen.كلاسات لكل جدول في قاعدة البيانات. يعرف نقاط الـ API. يدير البيانات والقواعد. لا يعلم شيئاً عن شكل الواجهة.
V
View — the user interfaceView — واجهة المستخدم
What the user sees. Displays data from Model. Sends user actions to Controller. Contains NO business logic.ما يراه المستخدم. يعرض بيانات Model. يرسل أفعال المستخدم للـ Controller. لا يحتوي على منطق عمل.
C
Controller — the middlemanController — الوسيط
Sits between Model and View. Receives input, validates it, updates Model, tells View to refresh.يجلس بين Model و View. يستقبل المدخلات، يتحقق منها، يحدّث Model، يأمر View بالتحديث.
Design patterns used inside MVCأنماط التصميم المستخدمة داخل MVC
Observer
Model (Subject) notifies View (Observer) on state change.Model (Subject) يُخطر View (Observer) عند تغيير الحالة.
Strategy
Controller handles different actions with interchangeable strategies.Controller يعالج الأفعال المختلفة باستراتيجيات قابلة للتبادل.
Singleton
One shared Model/Controller instance across the app.نسخة واحدة مشتركة من Model/Controller في التطبيق.
Composite
View: UI components nested inside each other.View: مكونات UI متداخلة داخل بعضها.
Factory Method
Controller creates View/Model without concrete types.Controller ينشئ View/Model دون الاعتماد على الكلاسات المحددة.
